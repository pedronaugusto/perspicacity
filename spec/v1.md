# Perspicacity v1 Specification

## Overview

Perspicacity is a structured scene perception format. It describes what an observer perceives in a 3D scene through a tiered model, from basic identity (what exists) through temporal changes (what's changing).

The canonical data model is **typed structs**. The text DSL (`.picacia`) is one serialization format optimized for LLM consumption. This spec defines both.

---

## Tier Model

```
Tier 0  IDENTITY    What exists         CAM, LIGHT, OBJ, SGROUP, ASSEMBLY, FOCUS, SCENE
Tier 1  SPATIAL     Where things are    REL, VERIFY, CONTAIN, SPATIAL
Tier 2  VISUAL      What camera sees    COMP, RAY, MVIEW
Tier 3  PHYSICAL    Light + physics     LIT, SHAD, MAT, HARMONY, PALETTE, WORLD, PHYS, CONTACT
Tier 4  SEMANTIC    What things mean    HIER, GRP
Tier 5  TEMPORAL    What's changing     DELTA, ANIM
```

**SND** (audio) and **CONTACT** (physics contacts) are reserved for future use — not currently implemented.

Each tier builds on the previous. A tier-0 perception is valid on its own. Higher tiers add richer detail. Consumers request the tiers they need.

---

## Struct-First Design

The canonical data model is a tree of typed structs rooted at `ScenePerception`. Every field has a defined type, default value, and semantics.

**Why struct-first:**
- Text DSL is generated from structs, not the other way around
- The spec is unambiguous — struct fields have types, text lines can be ambiguous
- Other serializations (JSON, binary) can be added without changing the data model

**Text DSL is one serialization.** The `.picacia` text format is optimized for LLM consumption: token-efficient, prefix-scannable, human-readable. But it is not the source of truth — the struct is.

---

## ScenePerception Struct

```
ScenePerception {
    viewpoint: Viewpoint,          # who is perceiving (camera, agent, synthetic)
    viewpoint_position: Vec3,
    viewpoint_forward: Vec3,
    identity: IdentityTier,
    spatial: SpatialTier,
    visual: VisualTier,
    physical: PhysicalTier,
    semantic: SemanticTier,
    temporal: TemporalTier,
}
```

Where:

```
IdentityTier {
    scene_header: Optional[SceneHeader],
    cameras: list[CameraData],
    lights: list[LightData],
    objects: list[ObjectData],
    semantic_groups: list[SemanticGroupData],
    assemblies: list[AssemblyData],
    focus: Optional[FocusData],
}

SpatialTier {
    relationships: list[RelationshipData],
    verify: list[VerifyResult],
    containment: list[ContainmentData],
    spatial_facts: list[SpatialFact],
}

VisualTier {
    composition: Optional[CompositionData],
    ray_grid: Optional[RayGridData],
    multi_views: list[MultiViewData],
}

PhysicalTier {
    light_analyses: list[LightAnalysisData],
    shadow_analyses: list[ShadowAnalysisData],
    materials: list[MaterialPrediction],
    harmony: Optional[HarmonyData],
    palette: Optional[PaletteData],
    world: WorldData,
    physics_states: list[PhysicsState],
    contacts: list[ContactData],
    sounds: list[SoundData],
}

SemanticTier {
    hierarchy: list[HierarchyEntry],
    groups: list[GroupEntry],
}

TemporalTier {
    animations: list[AnimationState],
    deltas: list[str],                 # free-form change descriptions (computed by diffing previous perception)
}
```

## Line Types

### Tier 0 — Identity

#### SCENE

Scene summary header.

```
SCENE obj_count objects light_count lights energyW engine [ground_z=N]
```

Example:
```
SCENE 42 objects 3 lights 1200W BLENDER_EEVEE ground_z=0
```

#### CAM

Camera definition. One per viewpoint.

```
CAM name [x,y,z] focal_mm
```

Example:
```
CAM Camera [0,0,5] 50mm
```

#### LIGHT

Light source.

```
LIGHT name TYPE energyW [r,g,b] [x,y,z] [type-specific] [noshadow]
```

TYPE is one of: POINT, SUN, SPOT, AREA.

Type-specific fields:
- SPOT: `cone=X° blend=Y` — cone angle in degrees and edge softness (0-1)
- AREA: `shape Xm` or `shape XmxYm` — shape (square, rectangle, disk, ellipse) and dimensions

The `noshadow` flag is appended when the light has shadow casting disabled.

Example:
```
LIGHT Key POINT 500W [1.0,1.0,1.0] [3,2,4]
LIGHT Sun SUN 2.5W [1.0,0.95,0.9] [0,0,10]
LIGHT Spot1 SPOT 300W [1.0,0.9,0.8] [4,3,2] cone=45° blend=0.15
LIGHT Fill AREA 200W [1.0,1.0,1.0] [-2,1,3] rectangle 1.0mx0.5m noshadow
```

#### OBJ

Object perceived from the current viewpoint.

```
OBJ name [x,y,z] coverage% quadrant d=depth_m material [dim=[w,h,d]] [face=DIR] [lum=N] [transparent] [has_uv|no_uv] [flipped_normals=N%] [non_manifold=N] [inside=name] [contains:[names]]
```

Objects are sorted front-to-back by depth. Coverage is the percentage of screen space the object occupies. Quadrant describes position in the frame using a 3x3 grid: `top-left`, `top-center`, `top-right`, `mid-left`, `mid-center`, `mid-right`, `bot-left`, `bot-center`, `bot-right`.

Material format: for solid-color materials, uses the RGB-derived color name with properties in parens: `dark_brown(rough=0.75)`. For textured materials: `textured`. Color names are lowercase with underscores (e.g., `warm_brown`, `steel_gray`, `near_black`).

Optional fields:
- `dim=[w,h,d]` — world-space bounding box dimensions in meters. Camera-independent. Essential for scale reasoning.
- `face=DIR` — the face of the object most visible to camera (+X, -X, +Y, -Y, +Z, -Z)
- `lum=N` — rendered luminance (0.0=black, 1.0=fully bright). Derived from micro-render analysis.
- `transparent` — material has transmission > 0.5 or Glass BSDF. Critical for containment reasoning.
- `has_uv` / `no_uv` — UV layer presence. Check before texturing.
- `flipped_normals=N%` — percentage of sampled face normals pointing inward (mesh quality, top-coverage objects only)
- `non_manifold=N` — count of non-manifold edges (mesh quality, top-coverage objects only)
- `inside=name` — this object is contained inside the named object
- `contains:[names]` — this object contains the listed objects

Example:
```
OBJ Cube [0,0,0] 15% mid-center d=5.0m light_gray(rough=0.5) dim=[2,2,2] face=+Z lum=0.85 has_uv
OBJ Sphere [-2,0,0] 8% mid-left d=5.4m beige(metal=1.0,rough=0.2) dim=[1,1,1] face=+Z lum=0.72 has_uv
OBJ Bottle [0,1,0] 22% mid-center d=3.0m glass(ior=1.45) dim=[0.08,0.08,0.3] transparent has_uv contains:[Bulb,Palace]
OBJ Bulb [0,1,0] 0.5% mid-center d=3.1m gold(emit=5) dim=[0.02,0.02,0.04] has_uv inside=Bottle
```

#### SGROUP

Semantic group — aggregates similar objects into a single line.

```
SGROUP "name" [x,y,z] dim=[w,h,d] top=N material [facing=DIR] members=N
```

Example:
```
SGROUP "Dining Chairs" [2,1,0.4] dim=[3,2,0.8] top=0.8 wood facing=N members=4
```

#### ASSEMBLY

Co-located multi-type objects forming a functional unit.

```
ASSEMBLY "name" members=[A,B,C] center=[x,y,z] types=TYPE+TYPE
```

Example:
```
ASSEMBLY "Desk Lamp" members=[Lamp_Base,Lamp_Arm,Shade,Bulb] center=[2,1,0.8] types=MESH+LIGHT
```

### Tier 1 — Spatial

#### REL

Spatial relationship between two objects.

```
REL A->B distance direction [vertical] [overlap[=N%]] [aabb_overlap=N%] [contact] [occludes] [occ=N%]
```

Direction is camera-relative: `left`, `right`, `behind`, `in_front`, `above`, `below`.
Vertical qualifiers: `same_level`, `above`, `below`.
Optional flags: `overlap` (screen-space, bare or with `=N%`), `aabb_overlap=N%` (world-space AABB intersection volume as % of smaller object — camera-independent), `contact`, `occludes`.
Optional: `occ=N%` — how much A occludes B.

The arrow can be written as `->` or `→`.

Example:
```
REL Cube->Sphere 2.0m left same_level
REL Cube->Floor 1.0m below contact aabb_overlap=5%
REL Crate_B->Crate_A 0.5m above overlap=65.2% aabb_overlap=12% contact
```

#### VERIFY

Post-modification verification. Only emitted after modifying tool calls, and only on failure. Reports mechanical failures where a tool reported success but the mesh didn't actually change. Silent on success.

```
VERIFY FAIL object_name reason
```

Example:
```
VERIFY FAIL Chair_0 parent moved but mesh AABB unchanged
```

#### CONTAIN

Containment relationship — one object is inside another.

```
CONTAIN outer contains inner full|partial
```

Full means all 6 axis-aligned rays from the inner object's center hit the outer object. Partial means 4-5 out of 6 hit.

Example:
```
CONTAIN Bottle contains LightBulb full
```

#### SPATIAL

Objective spatial facts about the scene. No judgment — consumers decide if a fact is a problem.

```
SPATIAL object fact_type key=value [key=value ...]
```

Fact types:

| Fact Type | Description | Detail Fields |
|-----------|-------------|---------------|
| `bbox_below_surface` | Object AABB extends below a surface | surface, surface_z, penetration, pct |
| `bbox_extends_into` | Object AABB overlaps another object | other, overlap_z, pct |
| `scale_diagonal` | AABB diagonal is extreme (>50m or <0.005m) | diagonal |
| `scale_ratio` | Non-uniform scale ratio >10:1 | ratio, axes |
| `no_material_slots` | Visible object has zero material slots | (none) |
| `near_plane` | Object bbox intersects camera near clip plane | dist, clip_start |
| `inside_bbox` | Light source inside object AABB | container, light_type |
| `no_ground_below` | Downward raycast finds no surface | cast_dist |
| `no_light_sources` | Scene has no lights, no HDRI, no emissives | lights, hdri, emissive |
| `energy_zero` | Light has zero energy | (none) |
| `off_camera` | Object has 0% screen coverage | (none) |
| `zero_dimensions` | Object has zero-volume AABB | zero_axes |

Example:
```
SPATIAL Bottle bbox_below_surface surface=TableTop surface_z=0.8 penetration=0.32 pct=46
SPATIAL GiantCube scale_diagonal diagonal=173
SPATIAL scene no_light_sources lights=0 hdri=false emissive=0
SPATIAL PointLight inside_bbox container=Room light_type=POINT
```

SPATIAL lines are always emitted when facts exist. An empty set of SPATIAL lines means no spatial anomalies detected.

SPATIAL is tier 1 (Spatial), alongside REL. CONTACT is deferred to v2 implementation (currently specced but unimplemented in the reference implementation).

### Tier 2 — Visual

#### COMP

Composition analysis of the current frame.

```
COMP thirds=score visible/total balance=X depth=D/3 [edge:[names]]
```

- `thirds` — rule-of-thirds alignment score (0.0 to 1.0)
- `visible/total` — number of objects visible vs total in scene
- `balance` — visual weight balance (0.0 = left-heavy, 1.0 = right-heavy, 0.5 = balanced)
- `depth` — depth layers used (1/3, 2/3, 3/3)
- `edge` — objects touching frame edge (optional)

Example:
```
COMP thirds=0.71 3/3_visible balance=0.62 depth=1/3
```

#### RAY

Ray-grid sampling of visible coverage.

```
RAY NxN Object=coverage% [empty=X%]
```

N is the grid resolution. Each object's coverage is the percentage of rays that hit it.

Example:
```
RAY 12x12 Cube=15% Sphere=8% Floor=35% empty=42%
```

#### MVIEW

Multi-view coverage from a named viewpoint.

```
MVIEW view: Object=coverage%
```

Example:
```
MVIEW front: Cube=20% Sphere=12% Floor=40%
MVIEW top: Cube=8% Sphere=6% Floor=70%
```

### Tier 3 — Physical

#### LIT

Light incidence analysis on a surface.

```
LIT light->surface @angle° i=intensity [eff=X] [raw=X] [shadow:casters]
```

The arrow can be written as `->` or `→`. `i=` is normalized intensity (0-1, relative to scene maximum). `eff=` is effective brightness on the absolute perceptual scale (same reference as OBJ lum=). `raw=` is absolute intensity (energy × cos(angle) / distance²).

The light source can be a scene light name or `EMIT:ObjectName` for emissive mesh objects acting as light sources.

Example:
```
LIT Key->Cube @42° i=0.85 eff=6.7 raw=372.5
LIT Key->Floor @65° i=0.45 eff=3.56 raw=198.1 shadow:Cube,Sphere
LIT EMIT:LightBulb->TableTop @25° i=0.45 eff=0.3
```

#### SHAD

Shadow analysis.

```
SHAD light->surface coverage% [casters:A,B] [contact|gap=Xm]
```

Example:
```
SHAD Key->Floor 12% casters:Cube,Sphere contact
SHAD Sun->Wall 5% casters:Tree gap=0.3m
```

#### MAT

Material description as perceived.

```
MAT name: appearance [-- needs X] [warning]
```

The `--` separator introduces notes about what the material needs or warnings.

Example:
```
MAT Default: light grey matte
MAT Metal: polished metal -- needs env reflections
MAT Brick: red brick wall -- needs normal map
```

#### WORLD

World/environment settings.

```
WORLD bg=[r,g,b] strength=X
WORLD hdri strength=X
```

Example:
```
WORLD bg=[0.05,0.05,0.05] strength=1.0
WORLD bg=[1,0.7,0.4] strength=0.05
```

#### HARMONY

Material distribution summary.

```
HARMONY types=distribution temp=temperature
```

Temperature is one of: warm, cool, neutral, mixed.

Example:
```
HARMONY types=wood+metal+concrete temp=warm
```

#### PALETTE

Scene color palette from micro-render analysis.

```
PALETTE [lum=N] color1 color2 ...
```

`lum=` is overall rendered luminance (0.0-1.0). Colors are space-separated lowercase names.

Example:
```
PALETTE lum=0.45 warm_brown dark_grey steel_blue
```

#### PHYS

Physics state of an object.

```
PHYS name type mass=Xkg [vel=[x,y,z]] [sleeping]
```

Type is one of: `static`, `dynamic`, `kinematic`.

Example:
```
PHYS Crate dynamic mass=15.0kg vel=[0.0,0.0,-1.2]
PHYS Floor static mass=0.0kg sleeping
```

#### CONTACT

Contact point between two objects (physics detail beyond REL's boolean contact flag).

```
CONTACT A<>B normal=[x,y,z] force=XN surface=material
```

Example:
```
CONTACT Crate<>Floor normal=[0,0,1] force=147.1N surface=concrete
```

#### SND

Sound source perception (game-engine only).

```
SND source type=ambient|point|directional vol=X dist=Xm dir=direction occ=X
```

- `type` — ambient (everywhere), point (localized), directional
- `vol` — perceived volume (0.0 to 1.0)
- `dist` — distance from listener
- `dir` — perceived direction (camera-relative)
- `occ` — occlusion factor (0.0 = clear, 1.0 = fully blocked)

Example:
```
SND Radio type=point vol=0.6 dist=3.2m dir=left occ=0.4
SND Wind type=ambient vol=0.3 dist=0.0m dir=above occ=0.0
```

### Tier 4 — Semantic

#### HIER

Object hierarchy chain (child to root).

```
HIER child > parent > grandparent
```

Example:
```
HIER Wheel_FL > Axle_Front > Car_Body
HIER Gun > Hand_R > Armature
```

#### GRP

Collection/group membership.

```
GRP collection_name: obj1, obj2, obj3
```

Example:
```
GRP Vehicles: Car_Body, Truck
GRP Props: Barrel, Crate, Lamp
```

### Tier 5 — Temporal

#### ANIM

Animation state of an object.

```
ANIM name action=Name frame=N/total [playing|stopped]
```

Example:
```
ANIM Character action=Walk frame=12/30 playing
ANIM Door action=Open frame=24/24 stopped
```

#### DELTA

Free-form change descriptions computed by diffing the current perception against the previous one. Not stored in the struct — generated on-the-fly by comparing consecutive `ScenePerception` snapshots.

```
DELTA description
```

Example:
```
DELTA Crate moved [0.5,0,0] (+X)
DELTA Key light energy 500→800W
DELTA Guard.base_color changed
```

DELTA lines are only emitted when perception is compared to a previous snapshot. A standalone `.picacia` file will not contain DELTA lines.

## Extended Senses — What's In and What's Out

### CUT: THREAT

Threat detection (hostile actors, environmental hazards) is derivable from REL + PHYS + OBJ. An agent that knows positions, velocities, and object types can compute threats. Adding a dedicated line type would duplicate data and add ambiguity about the boundary between perception and reasoning.

### KEPT: CONTACT

CONTACT carries physics detail beyond REL's boolean `contact` flag — specifically the contact normal, force magnitude, and surface material. This data is expensive to derive and useful for physics-aware AI (foot placement, collision response, surface sound selection).

### CUT: MEM

Memory is not perception. An agent's memory of previously perceived objects is a higher-level system built on top of perception snapshots. Perspicacity describes what is perceived *now*.

### CUT: PROP

Game-specific object properties (health, ammo, inventory) were considered for v1 but cut. There is no reference implementation to validate the design against, and the design space is too game-specific. May return in v2 with a generic key-value approach.

### RESERVED: SND

Sound is a genuinely different sense — an agent behind a wall can hear but not see. SND is reserved for future use when an implementation exists to validate the design.

## Design Principles

1. **Token efficiency** — every character earns its place. Short prefixes, no XML verbosity, no redundant labels.

2. **Prefix-scannable** — an LLM can skip to the section it needs by scanning for the 3-5 character prefix (CAM, OBJ, REL, LIT, etc.).

3. **Front-to-back depth sort** — OBJ lines are sorted by depth (nearest first). The most important objects come first.

4. **Camera-relative directions** — "left" means left in the camera frame, not world-space -X. This matches how humans and LLMs reason about scenes.

5. **Always-on, not on-demand** — in BlenderWeave, perception is auto-attached to every modifying command. AI reads the latest snapshot whenever it needs spatial awareness.

6. **Toggleable sections** — consumers request tiers 0-N. A quick check might need tiers 0-1. A full audit needs tiers 0-5.

7. **Struct-first, text is just one serialization** — the canonical data model is typed structs. The text DSL is one serialization optimized for LLMs.
